
# 🧠 Pico2（RP2350）で PRA32‑U2/M を 4 インスタンス同時に使えない理由

Pico2（RP2350）上で **PRA32‑U2/M を 4 インスタンス同時にリンク・実行することは物理的に不可能**です。  
これはコードの問題ではなく、**ハードウェアのメモリ構造そのものが原因**です。

---

## 📌 1. ビルド時に発生するエラー

PRA32‑U2/M を 4 つリンクしようとすると、以下のエラーが発生します：

```
cannot move location counter backwards (from 20093d94 to 20080000)
collect2.exe: error: ld returned 1 exit status
```

これはリンカが **SRAM の割り当て領域を超えたため、  
変数を配置できなくなった**ことを示しています。

---

## 📌 2. PRA32‑U2/M のメモリ使用量

PRA32‑U2/M は高機能なシンセエンジンであり、内部に多数の 32bit バッファを持ちます：

- オシレーター波形バッファ  
- フィルタバッファ  
- FX バッファ  
- モーフィングバッファ  
- LFO / モジュレーションバッファ  
- 内部ミックスバッファ  

これらにより、**1 インスタンスあたり 約 20〜24KB の RAM** を消費します。

---

## 📌 3. Pico2（RP2350）の RAM 制約

Pico2 の Core1 に実質的に割り当てられる SRAM は **約 48KB** です。  
（実際にはスタックや I2S バッファも必要なので、さらに少なくなります）

しかし PRA32‑U2/M を 4 つ使うと：

| パート | RAM 使用量 |
|-------|------------|
| A     | 約 20KB    |
| B     | 約 20KB    |
| C     | 約 20KB    |
| D     | 約 20KB    |
| **合計** | **約 80KB** |

→ **Core1 の 48KB を大幅に超過**

そのため、**物理的にメモリに収まらない**状態になります。

---

## 📌 4. 実際に起きる現象

PRA32‑U2/M を 4 つ使おうとすると、以下の順で問題が発生します：

1. **3 インスタンス（A+B+C）までは正常動作**  
2. 4 つ目（D）を追加すると：  
   - 人の声のようなノイズ  
   - 意味不明な発振  
   - 一部パートが鳴らない  
   → **RAM 破壊による動作不良**
3. さらに構成を重くすると：  
   - **リンクエラー（cannot move location counter backwards）**  
   → もはやバイナリが生成できない

これは **RAM オーバーフロー → メモリ破壊 → 最終的にビルド不能**  
という典型的な流れです。

---

## 📌 5. まとめ

- Pico2（RP2350）の Core1 に割り当てられる SRAM は **約 48KB**  
- PRA32‑U2/M は **1 インスタンスあたり 約 20〜24KB**  
- 4 インスタンスでは **80KB 以上必要**  
- → **物理的に不可能**

そのため Pico2 で PRA32‑U2/M を使う場合は：

- **最大 3 インスタンスまで**  
- 4 パート目は **軽量シンセ（Lite版）に置き換える**  
- または **RP2040（Pico）に戻す**

といった設計上の制約を考慮する必要があります。

---

# 🧠 なぜ RP2040（Pico）では PRA32‑U を 4 インスタンス動かせるのか？

RP2040 と RP2350（Pico2）は、同じ「256KB SRAM」を持つように見えるが、  
**内部構造（メモリマップ）がまったく違う**。

その違いが、PRA32‑U2/M のような「巨大な .bss を持つ DSP クラス」の動作に  
決定的な影響を与えている。

---

## 📌 1. RP2040 は「フラットな 264KB SRAM」を持つ

RP2040 の SRAM は次のように **1 本の大きな連続領域**として扱える：

```
0x2000_0000 ──────────────────────────────── 264KB 連続
```

このため：

- PRA32‑U（1 インスタンス ≈ 20〜24KB）
- ×4 インスタンス（≈ 80〜96KB）

を置いても、**連続したメモリ領域に収まる**。

さらに：

- I2S バッファ  
- Core1 スタック  
- KOSMOS2 の変数  

を足しても、まだ余裕がある。

つまり RP2040 は **「大きな箱に全部入る」**。

---

## 📌 2. RP2350（Pico2）は SRAM が「分割されている」

Pico2 の SRAM はこうなっている：

```
Core0 SRAM: 約 48KB
Core1 SRAM: 約 48KB
USB/Peri SRAM: 別領域
Scratch X/Y: 別領域
```

つまり **Core1 が使えるのは約 48KB だけ**。

PRA32‑U を 4 つ置くと：

```
20KB × 4 = 80KB
```

→ **Core1 の 48KB を超えて物理的に入らない**

そのため：

- ノイズ  
- 発振  
- メモリ破壊  
- 最後にリンクエラー  

という流れになる。

RP2040 と違って、Pico2 は **「小さな箱が複数ある構造」**なので、  
大きな荷物（PRA32‑U2/M の巨大バッファ）を 4 つ入れることができない。

---

## 📌 3. RP2040 は「Core0/Core1 で SRAM を共有」できる

RP2040 の特徴：

- Core0 と Core1 が **同じ 264KB SRAM を共有**している  
- どちらの Core も **全領域にアクセス可能**

つまり PRA32‑U2/M の巨大バッファを  
**Core1 に全部置いても問題ない**。

---

## 📌 4. RP2350（Pico2）は「Core1 専用 SRAM が固定サイズ」

RP2350 の特徴：

- Core1 は **専用 SRAM（約 48KB）しか使えない**  
- Core0 の SRAM を Core1 が使うことはできない  
- Scratch X/Y も PRA32‑U2/M の巨大バッファには使えない

つまり PRA32‑U2/M のような **巨大な .bss を持つ DSP クラスは  
Core1 の 48KB に収まらない**。

---

## 📌 5. 結論

RP2040 では PRA32‑U を 4 インスタンス動かせるが、  
Pico2（RP2350）でPRA32-U2/Mを動かせない理由は次の通り：

- RP2040 は **264KB のフラットな連続 SRAM**  
- RP2350 は **Core1 が使える SRAM が約 48KB に制限されている**  
- PRA32‑U2/M は **1 インスタンスあたり 20〜24KB の巨大バッファを持つ**  
- 4 インスタンス（80〜96KB）は **Pico2 Core1 の SRAM に物理的に収まらない**
